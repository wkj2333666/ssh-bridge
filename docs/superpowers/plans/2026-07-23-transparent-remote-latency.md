# Transparent Remote Latency Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (recommended) to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make warm SSH-bridge requests spend only one persistent-session request round trip plus unavoidable local framing/capture work, while matching local Codex's trusted-filesystem behavior and retaining lexical/path/hash/mutation safety.

**Architecture:** Keep one Rust `HostSession` and framed POSIX dispatcher per configured alias. Cache the resolved SSH policy and capability probe after first host initialization; submit later `remote_run` and fixed operations directly through the existing session with no per-request `ssh -G`, root observation frame, or physical-root guard wrapper. Keep the configured root as a lexical path boundary and use connection-time capability root metadata only as diagnostic context.

**Tech Stack:** Rust 2021, Tokio, existing framed SSH dispatcher, existing shell quoting/path validators, deterministic shell fixtures, Cargo unit/integration/release tests. No new Python runtime, dependency, or benchmark helper.

## Global Constraints

- Production implementation and new performance tests must be Rust; do not add Python code or a Python dependency.
- `host.profile.root` remains the lexical configured-root boundary; `RemotePath::resolve`, `pin_fixed_inputs`, expected-content hashes, atomic writes, output limits, cancellation, and mutation-unknown semantics remain enabled.
- A warm request sends exactly one `SessionRequest` to the persistent host session; no `CODEX_SSH_ROOT_OBSERVE` frame and no in-command physical-root guard are generated by default.
- `Capability.physical_root`, `root_device`, and `root_inode` remain connection-time diagnostic fields only; they are not authorization or retry conditions.
- Bash remains the default `remote_run` shell; an unavailable Bash is reported so the Agent can explicitly retry with `sh`.
- Preserve the public `elapsed_ms` field and document it as bridge operation time, not remote CPU/disk time; timing tests must separately measure local setup, session round trip, and remote command body where possible.
- Keep the existing per-host/global semaphores, request-level cancellation, output quotas, MCP framing, and one-session-per-alias concurrency model unchanged.

## Files and Responsibilities

- Modify `src/ssh/process.rs`: cache resolved `SshPolicy`, remove root-observation/guard code paths, render direct fixed/run commands, simplify fixed result types, and add Rust-only timing hooks used by tests.
- Modify `src/ssh/mod.rs`: stop re-exporting removed root identity types and expose the direct fixed-command length helper needed by search framing.
- Modify `src/ssh/dispatcher.sh`: remove test-only root-observation classification/rewriting so the dispatcher has no special root frame path; retain normal framed request, cancellation, quotas, and process-group handling.
- Modify `src/remote/read.rs`, `src/remote/write.rs`, `src/remote/patch.rs`, `src/remote/search.rs`, and `src/remote/metadata.rs`: remove cross-stage `RootIdentity` plumbing and use the direct fixed request contract.
- Modify `tests/ssh_transport.rs`, `tests/remote_ops.rs`, `tests/fixtures/fake-ssh.sh`, and `tests/dispatcher.rs`: assert one warm request frame, one first-time probe/identity setup, ordinary symlink-retarget semantics, and unchanged mutation/cancellation guarantees.
- Modify `tests/performance_acceptance.rs` and `tests/ssh_transport.rs`: report first/warm bridge/session timing and enforce no extra setup frame.
- Modify `README.md`, `docs/security.md`, `docs/performance.md`, `skills/remote-ssh-ops/SKILL.md`, and `skills/remote-ssh-ops/references/operations.md`: describe connection-time diagnostics, near-zero warm bridge overhead, and ordinary remote filesystem semantics.
- Update `docs/superpowers/specs/2026-07-23-transparent-remote-latency-design.md` only when implementation evidence requires clarifying wording; do not weaken the approved boundary.

- [ ] **Step 2: Implement cached policy initialization.** Under the existing per-host initializer lock, use this control flow in `initialize_host`:

```rust
if let (Some(policy), Some(capability)) = (
    self.policies.lock().await.get(host).cloned(),
    self.capabilities.get(host).await,
) {
    return Ok(((*policy).clone(), capability));
}
let policy = match self.policies.lock().await.get(host).cloned() {
    Some(policy) => (*policy).clone(),
    None => {
        let resolved_identity = self.resolve_identity_once(host, connect_timeout_ms, cancel).await?;
        let identity = self.record_or_reject_identity(host, resolved_identity).await?;
        let policy = SshPolicy::for_host(&self.config, self.config.host(host)?, &self.runtime, &identity)?;
        self.policies.lock().await.insert(host.to_owned(), Arc::new(policy.clone()));
        policy
    }
};
let capability = self.capabilities.get_or_probe(host, || async {
    self.probe_capability(&policy, host, root, connect_timeout_ms, cancel).await
}).await?;
Ok((policy, capability))
```
